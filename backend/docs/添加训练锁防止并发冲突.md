# 添加训练锁防止并发冲突

## 问题

用户询问：现在计算和训练如果有多人同时进行，会不会卡死？

## 分析结果

### 不会卡死，但会有问题

✅ **好消息**：
- 系统使用FastAPI异步处理，不会卡死
- 不同租户之间有文件夹隔离，不会冲突

❌ **坏消息**：
- 同一租户并发训练会导致文件冲突
- 历史记录可能被覆盖
- 结果不可预测

## 解决方案

添加训练锁机制，防止同一租户并发训练。

### 实现方式

#### 1. 添加全局训练锁字典

```python
# app/main.py 第57-60行
# 训练锁字典：防止同一租户并发训练
import threading
from collections import defaultdict
_training_locks = defaultdict(threading.Lock)
```

#### 2. 在训练开始时获取锁

```python
# app/main.py 第93-105行
# 获取租户的训练锁
lock = _training_locks[tenant_id]

# 尝试获取锁（非阻塞）
if not lock.acquire(blocking=False):
    logger.warning(f"租户 {tenant_id} 正在训练中，拒绝新的训练请求")
    raise HTTPException(
        status_code=409,
        detail=f"租户 {tenant_id} 正在训练中，请等待当前训练完成后再试"
    )
```

#### 3. 在训练结束时释放锁

```python
# app/main.py 第217-220行
finally:
    # 释放训练锁
    lock.release()
    logger.info(f"租户 {tenant_id} 训练锁已释放")
```

#### 4. 异常处理

```python
# app/main.py 第222-229行
except HTTPException:
    # HTTPException直接抛出（包括409冲突错误）
    raise
except Exception as e:
    logger.error(f"训练失败: {e}", exc_info=True)
    logger.error(f"错误类型: {type(e).__name__}")
    logger.error(f"错误字符串表示: {repr(e)}")
    raise HTTPException(status_code=500, detail=f"训练失败: {str(e)}")
```

### 修改的接口

1. **`/api/train`** - 普通训练接口
2. **`/api/train/stream`** - 流式训练接口

两个接口都添加了相同的锁机制。

## 技术细节

### 锁的类型

使用 `threading.Lock`：
- ✅ 线程安全
- ✅ 非阻塞获取（`acquire(blocking=False)`）
- ✅ 自动管理（使用finally确保释放）

### 为什么使用非阻塞锁？

```python
# 非阻塞（推荐）
if not lock.acquire(blocking=False):
    raise HTTPException(status_code=409, ...)

# 阻塞（不推荐）
lock.acquire(blocking=True)  # 会一直等待，可能导致请求超时
```

**优点**：
- ✅ 立即返回409错误
- ✅ 用户体验好（知道需要等待）
- ✅ 不会占用服务器资源

### 锁的生命周期

```
请求开始
  ↓
尝试获取锁
  ↓
获取成功？
  ├─ 是 → 执行训练 → 释放锁 → 返回结果
  └─ 否 → 返回409错误
```

### 异常安全

使用 `try...finally` 确保锁一定会被释放：

```python
try:
    # 执行训练
    ...
finally:
    # 无论成功还是失败，都释放锁
    lock.release()
```

## 限制和注意事项

### 1. 单进程有效

这个锁机制只在单进程内有效。如果使用多个uvicorn worker，需要使用其他方案：

```bash
# 单进程（当前方案有效）
uvicorn main:app --host 0.0.0.0 --port 8000

# 多进程（当前方案无效）
uvicorn main:app --host 0.0.0.0 --port 8000 --workers 4
```

**解决方案**：
- 使用Redis锁（分布式锁）
- 使用文件锁（跨进程）
- 使用数据库锁

### 2. 锁粒度

当前锁的粒度是**租户级别**：
- ✅ 同一租户：串行执行
- ✅ 不同租户：并行执行

### 3. 锁泄漏风险

如果程序崩溃，锁可能不会被释放。但由于使用 `threading.Lock`，进程重启后锁会自动清除。

## 性能影响

### 对单个请求

- ✅ 几乎无影响（获取锁只需几微秒）

### 对并发请求

- ✅ 不同租户：无影响，可以并发
- ⚠️ 同一租户：串行执行，第二个请求会被拒绝

## 未来改进

### 1. 添加队列机制

当前方案：第二个请求直接被拒绝

改进方案：第二个请求进入队列等待

```python
# 伪代码
if not lock.acquire(blocking=False):
    # 加入等待队列
    await wait_in_queue(tenant_id)
```

### 2. 添加超时机制

如果训练任务卡住，锁永远不会释放。

改进方案：添加超时自动释放

```python
# 伪代码
lock.acquire(timeout=3600)  # 1小时超时
```

### 3. 使用分布式锁

如果需要多进程部署，使用Redis锁：

```python
import redis
from redis.lock import Lock

redis_client = redis.Redis(host='localhost', port=6379)
lock = Lock(redis_client, f"training_lock:{tenant_id}", timeout=3600)

if not lock.acquire(blocking=False):
    raise HTTPException(status_code=409, ...)
```
